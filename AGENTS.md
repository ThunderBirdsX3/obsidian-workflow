# AGENTS.md — obsidian-workflow

AI agent guide for working on this repository.

> **This repo is the SOURCE of the toolkit — not a user project.**
> Do NOT run `/ow-*` commands here. Edit files directly.
> See [toolkit-builder memory](https://github.com) note in CLAUDE.md.

## What this repo is

`obsidian-workflow` is a spec-driven development toolkit that Claude Code projects install via `install.sh`. It provides:

- 22 slash commands (`/ow-*`) as `.claude/commands/` shims
- 9 specialized subagents in `.claude/agents/`
- Obsidian vault skeleton at `docs/vault/`
- Config resolution via `scripts/ow-paths.sh` (requires `yq`)

## Key structure

```
.ow/commands/        full command specs — source of truth for slash commands
.claude/commands/    GENERATED shims (point to .ow/commands/) — do not hand-edit
.claude/agents/      subagent definitions — model pinned from .ow.yml via manifest script
scripts/
  ow-paths.sh        resolves all vault/config paths — every command sources this
  ow-claude-manifest.sh  regenerates .claude/ from .ow.yml (--apply flag)
  ow-upgrade.sh      upgrades an installed project (--dry-run / --rollback)
  test.sh            smoke tests — run to verify toolkit integrity
install.sh           copies toolkit into a target project
.ow.yml              project config (vault_path, subagents, commands model, guardrails)
docs/vault/          Obsidian vault skeleton (00-Index … 95-Handoff)
```

## How to develop

### Edit a command

Edit `.ow/commands/ow-<name>.md` directly.
The `.claude/commands/ow-<name>.md` shim auto-delegates there — no sync needed for content.

### Add a new command

1. Create `.ow/commands/ow-<name>.md`
2. Add shim to `.claude/commands/ow-<name>.md` (copy pattern from existing shim)
3. Register in `.ow.yml` if model-pinnable
4. Add to command table in README.md and CLAUDE.md

### Change agent model or add an agent

Edit `.ow.yml` `subagents:` block, then run:

```bash
bash scripts/ow-claude-manifest.sh --apply
```

Do NOT hand-edit frontmatter in `.claude/agents/` — next sync overwrites it.

### Run smoke tests

```bash
bash scripts/test.sh
```

Tests verify: path resolution, config parsing, shim integrity, install/upgrade script.

### Test install flow

```bash
# Direct install (no prior clone)
TMP=$(mktemp -d)
git init "$TMP"
bash install.sh "$TMP"
ls "$TMP/.ow/" "$TMP/.claude/"
rm -rf "$TMP"

# Bootstrap script (simulates end-user curl flow)
TMP2=$(mktemp -d)
git init "$TMP2"
bash bootstrap.sh "$TMP2"
rm -rf "$TMP2"
```

## Critical rules for AI agents

| Rule | Why |
|---|---|
| Don't run `/ow-*` commands in this repo | Commands call `ow-paths.sh` expecting a USER project layout — it breaks here |
| Don't hand-edit `.claude/commands/` shims | Generated by `ow-claude-manifest.sh` — manual edits get overwritten |
| Don't hand-edit `.claude/agents/` model frontmatter | Same — sync overwrites it |
| Path resolution always through `scripts/ow-paths.sh` | Hardcoded paths = broken installs |
| `CLAUDE.md` is shipped to user projects | Keep it project-agnostic; no toolkit-internal paths |

## install.sh behaviour

`install.sh <target-dir>` copies:
- `.ow/` (commands, policies, checklists, templates, rules)
- `.claude/` (shims, agents)
- `scripts/`
- `docs/vault/` skeleton
- `.ow.yml` (default config)
- `CLAUDE.md`

Does NOT overwrite: existing `.ow.yml` values, existing `.ow/rules/` content, existing vault notes.

## upgrade.sh behaviour

`ow-upgrade.sh <target-dir>` refreshes toolkit-owned files only.
User-customised files are left untouched. Backups auto-created before overwrite.
`--dry-run` shows diff without applying. `--rollback` restores last backup.
